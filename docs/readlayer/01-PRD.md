# PRD：ReadWeave（织读）

> 文档状态：Proposed
> 产品阶段：需求核对
> 需求基线：PRD-0.1 / 2026-07-14

## 1. 背景与问题

当前学习流程的有效部分是“阅读者在原文现场持续提出问题，并把回答和术语解释放回上下文附近”。真正的成本不在提出问题，而在以下机械环节：

1. 离开 Trilium，把问题和上下文复制到 Codex、DeepSeek 或其他模型。
2. 等待回答后再复制回来，寻找原位置。
3. 手动加粗、取消加粗、缩进、着色和整理长回答。
4. 无法在提问前可靠发现已问过的相似问题或已定义的术语。
5. 同一术语/问答被复制到多篇文章后产生多个版本；修改一处不能传播到其他文章。
6. 旧 ReadLayer 尝试提前猜测用户问题并生成大量内容，速度慢、噪声高，也违背真实学习路径。

## 2. 产品愿景

让阅读者始终停留在 Trilium Web 的文章和自己的思考里：在原文段落主动提出一个真实问题，系统选择“最小但充分”的上下文、调用联网模型、查重并生成单个答案；只有用户审核提交后，答案才成为具有不可变 ID 的索引对象，并通过文章 ID、锚点 ID 和对象 ID 在所有位置显示最新内容。

## 3. 产品原则

| ID | 原则 |
|---|---|
| P-001 | 人发起问题，AI 不抢先批量替人决定要学什么。 |
| P-002 | 优先消灭复制、粘贴、排版、定位和查重，而不是追求表面上的全自动。 |
| P-003 | 文章是阅读现场，问答/术语是索引对象；二者只通过不可变 ID 关联，不按名称绑定，不复制真相。 |
| P-004 | 默认本地优先、用户拥有数据；模型调用范围和外发内容必须可见、可控。 |
| P-005 | 所有回答规则和显示设置只能由用户显式修改；应用不自动学习偏好。 |
| P-006 | 所有 AI 写入可追溯、可撤销；模型输出不是未经确认的事实。 |
| P-007 | 索引可以重建，知识不能依赖索引才能存在。 |
| P-008 | 尽量复用上游能力并保持可升级，核心魔改必须有明确收益和测试保护。 |
| P-009 | 默认一问一答，不把多轮聊天记录当作产品数据结构。 |
| P-010 | 上下文选择过程版本化、可复查，并以“满足回答所需的最小上下文”为目标。 |

## 4. 目标用户与核心任务

### 4.1 首要用户

重度个人学习者：长期在 Trilium Web 中阅读技术文章、论文、书籍或资料；一篇文章会产生几十到上百个问题；重视段落锚点、稳定索引、长期积累和跨文章复用。

### 4.2 可扩展用户

- 研究者：对来源、引用和概念演化有更高要求。
- 工程师：围绕代码、规范和术语形成可更新知识卡。
- 学生：需要不同深度、例子和先修知识解释。

### 4.3 Jobs to be Done

| ID | 当…… | 我想要…… | 从而…… |
|---|---|---|---|
| JTBD-001 | 阅读到不理解或值得追问的段落时 | hover/点击段落并直接提问 | 保持思考连续性 |
| JTBD-002 | 遇到术语时 | 一键得到符合我偏好的定义并保存 | 不再手动搬运与排版 |
| JTBD-003 | 准备提问或定义时 | 先看到已存在的相似知识 | 避免重复劳动，同时保留提出不同问题的自由 |
| JTBD-004 | 同一知识出现在多篇文章时 | 让多个锚点连接同一索引对象 ID | 修改一次，所有位置更新 |
| JTBD-005 | AI 回答不合适时 | 修改、重试、补证据或创建上下文变体 | 让知识逐步贴合个人理解 |
| JTBD-006 | 更换模型、设备或升级软件时 | 继续访问全部知识与引用 | 不被单一模型或索引锁定 |

## 5. 成功指标

### 5.1 北极星指标

`每篇文章中，用户从点击段落到审核提交一个可复用索引对象的中位主动操作次数`。

首发目标：常规问答或术语从选中段落到审核提交不超过 3 个明确动作；查看已有答案只需 hover/click，不要求展开文章内卡片。

### 5.2 结果指标

| ID | 指标 | Beta 目标 |
|---|---|---|
| M-001 | 不离开 Trilium Web 完成的问答比例 | ≥ 95% |
| M-002 | 保存后需要手工修复格式的比例 | ≤ 5% |
| M-003 | 被相似提示成功复用的条目比例 | 记录基线后设目标，不以越高越好 |
| M-004 | 索引对象修改后锚点显示一致性 | 100%，同一 Web 会话刷新 ≤ 1 秒 |
| M-005 | 含 100 个段落问题组的文章编辑数据丢失 | 0 |
| M-006 | AI 写入可由修订或撤销恢复的比例 | 100% |
| M-007 | 索引删除后可重建并恢复查重 | 100% |
| M-008 | 用户认为“系统替我乱做决定”的关键反馈 | 0 个未解决发布阻断项 |

### 5.3 反指标

- 不以自动生成问题数量作为成功指标。
- 不以单纯 token 消耗、对话长度或知识卡数量作为成功指标。
- 不鼓励为了提高复用率而把语义不同的问题强行合并。

## 6. 产品范围

### 6.1 P0：成熟首发必须具备

- Trilium Server/Web 在隔离数据目录中可部署、升级、备份和回滚。
- hover 段落显示可交互状态；点击即选择整段并显示该段全部问题标题。
- hover/focus 问题标题展开答案；点击可固定展开，保证触屏和键盘可用。
- 当前文章的单问单答界面，支持选择段落作为起始上下文。
- “提问”“定义术语”“引用已有知识”三种显式动作。
- 流式回答、停止、重试、编辑、审核提交、丢弃；不保留多轮聊天产品形态。
- 未提交答案仅停留在当前界面检查点，不进入索引库。
- 稳定 ID + JSON Schema 的问题、答案、术语、文章、锚点和连接索引对象。
- 精确重复和全文/模糊相似提示。
- 修改索引对象后所有连接锚点显示最新内容。
- 引用来源、创建/修改时间、模型和文章上下文可追溯。
- 本地优先、离线可读写已有知识；索引可重建。
- DeepSeek 官方兼容 API 作为第一 Provider，base URL 与模型名可配置。
- API 密钥只存在 Web 服务端 secret/env，不进入笔记、数据库、导出、日志或客户端 bundle。
- 修订、撤销、软删除、备份、损坏引用提示。
- 大文章与大量段落/问题索引的性能验证。
- 问题、答案、文章、锚点及 ID 连接的独立 JSON 导出。

### 6.2 P1：Beta 前完成

- 语义向量查重与可解释的相似原因。
- 显式、版本化、确定性的回答规则与显示设置。
- 导出时可选择“保持实时引用”或“展开为静态快照”。
- 锚点位置浏览、反向链接、批量合并与上下文变体。
- 模型成本、上下文范围和外发数据预览。
- 快捷键、命令面板与无鼠标流程。

### 6.3 P2：正式发布后候选

- 移动端完整编辑和问答。
- 以已导出索引为输入的知识图谱视图。

### 6.4 明确不做

- 默认扫描整篇文章并自动生成几十或上百个问题。
- 未经确认自动合并、覆盖或删除索引对象。
- 保存或展示多轮聊天历史作为产品形态。
- 依赖中心化 ReadLayer 云端才能阅读自己的内容。
- 任何本地/离线生成模型。
- 自动学习用户偏好、自动创建回答规则。
- 复习、测验和多人共享。
- 首发即支持所有模型、所有系统和所有 Trilium 插件组合。

## 7. 功能需求

优先级：`P0` 首发阻断，`P1` Beta 阻断，`P2` 后续候选。

### 7.1 阅读上下文与提问

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| FR-001 | 系统必须在 Trilium Web 文章内工作，不要求跳到外部聊天工具。 | P0 | 从选择到审核提交全程不离开浏览器应用。 |
| FR-002 | 用户点击段落后，系统以该段为起点自动选择“最小充分上下文”。 | P0 | 上下文按相邻段、章节、全文、关联对象逐级扩张；用户可查看实际范围。 |
| FR-003 | 用户可显式发起“提问”“定义术语”“索引已有”三种动作。 | P0 | 三条路径均从段落交互或命令触发。 |
| FR-004 | 切换文章后上下文和未提交界面检查点按 articleId 隔离。 | P0 | 不把上一篇文章上下文静默带入下一篇。 |
| FR-005 | 单个回答支持流式显示、停止、重生成和编辑。 | P0 | 不构建多轮聊天；停止后的内容仍留在当前界面。 |
| FR-006 | 用户可以在提交前编辑问题、答案、术语、别名和来源。 | P0 | 提交预览就是索引对象内容。 |
| FR-007 | 未经用户审核的回答不得保存到索引库。 | P0 | 未提交内容留在当前 Web 界面检查点；刷新/过期策略有明确提示。 |
| FR-008 | 发送前必须展示算法选定的段落、章节、附件、关联对象和网页搜索范围。 | P0 | 每项可查看；自动扩张过程可解释、可限制。 |
| FR-009 | 对超长上下文必须给出截断/摘要提示，而不是静默丢失。 | P0 | 可查看实际发送范围、扩张层级与估算 token。 |
| FR-010 | 上下文选择器必须版本化并进入评测 harness。 | P0 | 同一版本的处理步骤确定；升级可比较正确率、token 与回答质量。 |

### 7.2 索引对象、段落锚点与 hover 展示

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| FR-020 | 问答必须保存为独立索引对象，具有不可变 questionId；名称不是连接键。 | P0 | 改标题、移动或重名均不破坏连接。 |
| FR-021 | 术语定义必须保存为独立索引对象，具有不可变 termId 并支持别名。 | P0 | 两个同名术语可并存，hover 内容用于区分。 |
| FR-022 | 每个可提问段落具有稳定 anchorId，并与 articleId 共同定位。 | P0 | 编辑文章后可检测锚点漂移；连接不靠段落标题。 |
| FR-023 | hover 段落显示交互提示；点击段落即选择全段并列出该段全部问题标题。 | P0 | 无需插入可见问答卡片。 |
| FR-024 | hover/focus 问题标题展开答案；点击可固定，触屏有等价操作。 | P0 | 长答案不占用常驻文章空间。 |
| FR-025 | 用户可以从问题/术语展开层打开或编辑索引对象。 | P0 | 明示修改将传播到多少 articleId/anchorId。 |
| FR-026 | 用户可以将某个上下文派生为变体，而不是修改全局索引对象。 | P1 | 原对象和变体保留稳定 ID 关系。 |
| FR-027 | 每个对象记录来源 articleId、anchorId、原始摘录、创建方式和模型元数据。 | P0 | 来源可导航；摘录变更可检测。 |
| FR-028 | 只有提交后的问题/答案进入修订；界面检查点与已提交对象严格分离。 | P0 | 能区分未提交界面状态、人工编辑和已验证。 |
| FR-029 | 同一索引对象可以连接多篇文章和同一文章的多个锚点。 | P0 | 所有位置共用最新内容但保留各自显示状态。 |
| FR-038 | 所有缩写/英文名按 `[缩写 + 空格]中文全称（英文全称）` 输出；无缩写时省略缩写部分。 | P0 | 格式校验器和生成后验证均通过。 |

### 7.3 查重、相似与复用

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| FR-030 | 提交或发送前执行规范化精确查重。 | P0 | 文字大小写、常见标点、空白差异不造成重复。 |
| FR-031 | 提供全文与模糊相似候选。 | P0 | 初始列表只显示标题；hover/focus 后显示内容与来源。 |
| FR-032 | 提供可选语义相似检索，索引可重建。 | P1 | 关闭或损坏向量索引时核心流程仍工作。 |
| FR-033 | 相似提示不阻止用户提出或提交新问题。 | P0 | 用户可索引已有、建立变体或即使重复仍新建。 |
| FR-034 | 合并必须由用户确认，并保留来源、引用和修订历史。 | P1 | 合并后无悬空引用。 |
| FR-035 | 用户可以查看对象被哪些文章、哪些位置引用。 | P0 | 列表可导航并显示数量。 |
| FR-036 | 系统必须区分“字面相似”与“意图相同”，并展示理由。 | P1 | 至少标注标题/正文/别名/语义命中类型。 |
| FR-037 | 相似阈值、候选数量和是否自动展开必须可配置。 | P1 | 配置有安全默认值。 |

### 7.4 ID 连接与全局传播

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| FR-040 | 文章/连接索引只保存 objectId、articleId、anchorId 和显示元数据，不保存答案副本。 | P0 | 检查文章和 link 记录不存在重复答案正文。 |
| FR-041 | 修改索引对象后，所有 articleId/anchorId 连接立即读取最新内容。 | P0 | 当前 Web 视图刷新；重新打开必为最新内容。 |
| FR-042 | 索引对象被软删除、保护或不可用时显示可操作的损坏连接状态。 | P0 | 可恢复、替换、移除连接；不静默消失。 |
| FR-043 | 删除锚点连接不能误删索引对象；删除索引对象必须显示影响范围。 | P0 | 两类删除有不同确认文案。 |
| FR-044 | AI 或人工修改索引对象均产生可识别的修订来源。 | P0 | 可以恢复到修改前版本。 |
| FR-045 | 当前不实现跨设备同步；任何未来同步不得改变 ID 连接语义。 | P2 | 首发只验证备份、恢复和导出。 |

### 7.5 显式回答规则与显示设置

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| FR-050 | 用户可显式设置语言、深度、结构、举例、公式、代码和引用规则。 | P1 | 可查看、修改、版本化；系统不自动改变。 |
| FR-051 | 用户可为问答和术语分别配置确定性模板。 | P1 | 模板预览显示将发送给模型的系统规则。 |
| FR-052 | 项目/文章可显式覆盖全局规则，并清楚显示继承来源。 | P1 | 不出现不可解释的提示词变化。 |
| FR-053 | 系统不得从用户行为、编辑或接受记录自动学习偏好。 | P0 | 规则变化只来自显式设置操作，并有审计记录。 |
| FR-054 | 用户可导出、重置和版本化回答规则。 | P1 | 重置不修改索引对象。 |

### 7.6 Provider、模型与权限

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| FR-060 | DeepSeek 官方 API 是首发第一 Provider。 | P0 | 完成配置、流式、停止、错误、推理模式和模型变化测试。 |
| FR-061 | Provider 支持可配置 base URL/model，不硬编码可能废弃的模型名称。 | P0 | 使用官方兼容接口；模型名变化无需迁移知识数据。 |
| FR-062 | API 密钥仅由 Web 服务端 secret/env 提供，不传给浏览器。 | P0 | 客户端 bundle、网络响应、DB、导出、日志扫描无密钥。 |
| FR-063 | 模型笔记访问范围必须先通过 Trilium 原生权限/保护继承检查。 | P0 | ReadWeave 不创建第二套 ACL；发送前范围可见。 |
| FR-064 | 修改/创建/删除笔记等工具必须同时通过 Trilium 权限与逐次操作确认。 | P0 | 只读问答不能获得写入工具。 |
| FR-065 | 无 Provider、离线、限流或余额不足时，已有索引仍可读写和导出；不能生成。 | P0 | 错误不会破坏未提交界面内容。 |
| FR-066 | 记录模型、Provider、提示模板版本和生成时间，但不记录密钥。 | P0 | 可解释答案来源。 |
| FR-067 | Web 搜索必须可关闭，并对外发查询给出提示。 | P0 | 默认策略由用户确认。 |

### 7.7 导入、导出、备份与维护

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| FR-070 | 旧手工蓝/绿问答块迁移不进入当前范围。 | P2 | 只有未来出现明确需求时重新立项。 |
| FR-071 | 提供独立、版本化的 JSON 索引导出：objects、articles、anchors、links。 | P0 | 所有连接按 ID；不依赖名称；通过 JSON Schema 校验。 |
| FR-072 | 相似索引可手动检查、清空、暂停和重建。 | P0 | 重建失败不影响知识对象。 |
| FR-073 | 升级前自动创建独立备份并验证可读。 | P0 | 回滚演练通过。 |
| FR-074 | ReadWeave 真相数据必须随 Trilium 原生备份覆盖，派生索引可重建。 | P0 | 恢复后 ID 连接完整；当前不要求跨设备同步。 |
| FR-075 | 提供健康检查：悬空 link、孤立 anchor、重复 canonical key、索引漂移。 | P1 | 检查结果可修复或导出。 |

| ID | 需求 | 优先级 | 验收摘要 |
|---|---|---:|---|
| FR-080 | 高频操作提供可重映射快捷键和命令面板入口。 | P1 | 与 Trilium 现有快捷键冲突检测。 |
| FR-081 | 所有关键操作可仅用键盘完成。 | P1 | 键盘验收脚本通过。 |

| FR-083 | 错误提示必须说明数据是否已保存、如何重试和如何恢复。 | P0 | 不出现只写“失败”的死胡同。 |
| FR-084 | 提供可脱敏诊断包，默认排除笔记正文、密钥和完整提示。 | P1 | 用户可预览导出内容。 |

## 9. 关键验收场景

### AC-001：原位提问闭环

给定一篇 Trilium Web 文本笔记，用户 hover 某段看到交互提示并点击整段；界面列出该段已有问题标题；用户输入新问题，系统展示最小充分上下文和相似标题并生成单个答案；答案未经提交只留在界面；用户审核提交后生成 questionId 和 anchor link；再次 hover 该段可看到标题，hover/focus 标题可展开答案。

### AC-002：术语跨文章复用

给定文章 A 的 anchorId 已连接 termId=B，文章 C 再次遇到同名术语；候选初始只显示标题，hover 展示具体内容；用户按 ID 连接已有 B；随后修改 B，A/C 所有锚点都显示最新定义，且文章与 link 索引中不存在答案副本。

### AC-003：相似但不同的问题

用户提出与已有问题字面相似、意图不同的新问题；系统只显示候选标题并允许 hover 比较，不阻止提交；用户创建上下文变体或明确重复新建；两个 questionId 保持关联但答案互不覆盖。

### AC-004：离线与失败

生成过程中断网；已输入问题和当前流式内容仍留在界面检查点；用户可重试或丢弃，但未审核内容不进入索引库；已有锚点问答继续正常显示和编辑；相似向量服务不可用时退化到精确/全文候选。

### AC-005：删除与恢复

删除文章 C 的 anchor link 只删除连接，不删除索引对象 B；尝试删除 B 时显示引用数量与位置；确认软删除后连接显示损坏状态；从 Recent Changes 恢复 B 后连接自动恢复。

### AC-006：升级与回滚

在现有 0.101.3 数据副本上部署/升级 Web 服务前生成并校验备份；升级后文章、索引对象和锚点连接可读；模拟失败后按文档恢复备份并重新启动；不得用真实唯一数据做首次实验。

### AC-007：索引导出

导出生成通过 JSON Schema 的 bundle；任意 question/term 可通过 objectId 找到，任意 link 可通过 articleId + anchorId + objectId 还原连接；两个同名对象不会因名称相同发生混淆。

### AC-008：上下文选择器

在公开多源 corpus 中，对每个问题记录选择器版本、候选上下文层级、最终发送段落、token 和答案质量；选择器能在信息不足时逐级扩张，在段落已充分时不发送全文；升级版本可与旧版本端到端比较。

## 10. 依赖与假设

- 使用 TriliumNext v0.103.0 作为当前源代码基线；开发开始前再次确认是否已有更新稳定版。
- 首批样本来自 AMD Vivado、RFC、RISC-V、arXiv、Trilium 与中文 Python 官方资料，全部在隔离 Web/Server 数据目录验证。
- DeepSeek API 可用；已暴露密钥不使用，轮换后的新密钥只通过服务端 secret/env 注入。
- Trilium Anchor/relations/隐藏 JSON note 可支撑 ID 连接原型；最终需要段落 hover/anchor 插件，而不是常驻问答卡片插件。
- 当前不实现跨设备同步与多人共享，但必须完成备份、恢复和独立索引导出。
- LangChain/LangGraph 不是默认依赖；先用显式 TypeScript 状态机、JSON Schema、界面检查点和评测 harness，只有复杂工作流实验失败时再引入。

## 11. 发布判定

只有以下条件全部满足，才能称为“成熟产品发布”：

1. 所有 P0 需求为 `Verified`，不存在未说明的豁免。
2. AC-001 至 AC-008 在公开多源 corpus 和数据副本上通过。
3. 数据升级、备份、崩溃恢复和回滚完成演练。
4. 100 段落问题组长文、50,000 索引对象基准达到固定阈值。
5. 安全/隐私检查、依赖扫描、XSS 测试和密钥泄漏扫描通过。
6. AGPL 源码提供、归属、许可证和品牌说明通过发布检查。
7. 安装包、自动更新、数据目录隔离、卸载不删数据均已验证。
8. 已有至少一个完整 Beta 周期，且无 P0/P1 数据完整性缺陷。
